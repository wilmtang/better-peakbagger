# 3D map audit — findings and fix plan (2026-07-21)

An audit of the 3D terrain feature across its four runtime pieces: the
MapLibre frame (`src/terrain/terrain-frame.js`, hosted by `terrain/terrain.html`), the
isolated-world bridge (`src/terrain/terrain-map.js`), the three page-world
coordinators (`src/maps/big-map.js`, `src/maps/peak-map.js`, the terrain section of
`src/gpx/gpx-analyzer.js`), and the shared helpers (`terrain-basemap.js`,
`terrain-camera.js`, `terrain-compass.js`, `terrain-cache.js`,
`terrain-tiles.js`, `peak-markers.js`, plus the background `TERRAIN_PREFETCH`
handler). Baseline at audit time: `npm test` green, 545/545.

Two user-reported symptoms motivated the audit and both are reproduced from
the code below: the compass "jiggling 180°" during rotation, and 2D→3D with
OpenTopoMap selected hanging for a long time and then silently failing,
almost always on BigMap pages.

## Confirmed bugs

### B1 — Compass needle spins the long way when the bearing crosses north

The frame streams `map.getBearing()` (MapLibre range −180…180) once per
painted frame. Both coordinators that show the compass re-normalize it to
[0, 360) — `big-map.js` (`((data.bearing % 360) + 360) % 360`, in the `view`
handler) and `peak-map.js` (same expression) — and
`terrain-compass.js` applies it as
`rotateZ(${-bearing}deg)`. `terrain-map.css` puts
`transition: transform 0.12s ease-out` on the disc.

CSS interpolates the two transform lists per-function numerically. At the
north crossing the normalized bearing jumps between ≈0 and ≈360, so the
transition animates a ~358° rotation the long way around instead of the ~2°
short way — and because the stream restarts the transition every frame, the
needle visibly flips through south and back: exactly the reported "jiggle
180°". The south crossing (±180) is continuous after normalization, which is
why the jiggle appears only "in certain circumstances".

**Fix.** Unwrap the angle inside `terrainCompass.update()` so the applied
rotation is continuous: keep a cumulative rotation and add the shortest-arc
delta each update (`delta = ((target − current) mod 360 + 540) mod 360 − 180`).
The fix belongs in the shared compass module, not the two coordinators, so
both surfaces (and any future one) are covered at once; the coordinators'
[0, 360) normalization becomes harmless and can stay.

**Regression test.** Extend the existing compass tests in
`test/maps/big-map.test.mjs` / `test/maps/peak-map.test.mjs`: stream a bearing sequence
that crosses north (e.g. 350 → 10) and assert the emitted `rotateZ` angle
changes by less than 180° between consecutive updates.

### B2 — A slow drape (OpenTopoMap) blocks the 3D boot until timeout

`createTerrain()` in `terrain-frame.js` builds the MapLibre map with the
active drape **inside the constructor style** (`terrainStyle(activeTheme,
activeBasemap)`). MapLibre's `load` event does not fire until
`style.loaded()` is true, which requires every source's initial tiles for the
current viewport to be loaded **or errored** (verified in the vendored
maplibre-gl 5.24.0, `Map.loaded()`). Tiles that are merely slow — in flight,
not errored — block `load` indefinitely.

OpenTopoMap is the worst case by design: it is a throttled volunteer server,
our spec deliberately pins one subdomain (`a.tile.opentopomap.org`) and keeps
the stock LOD as a courtesy (`terrain-basemap.js`, `L_OT`). On BigMap the
viewport is full-screen and the camera starts at pitch 60, so the initial
drape set is dozens of tiles out to the horizon. That routinely exceeds the
frame's 15 s `MAP_LOAD_TIMEOUT_MS` → `fail('timeout')` → the bridge tears the
frame down and posts `error` to the page → the coordinator reverts to 2D.
The ascent analyzer's smaller panel needs fewer tiles, which is why the hang
is "almost every time on BigMap" and rarer elsewhere.

**Fix.** Never let the drape gate the boot: construct the map with the
terrain-only style (`terrainStyle(theme, null)`) and add the drape
immediately after `load` using the machinery that already exists for live
picker swaps (`addBasemapLayer()` + `applyBasemapLod()`). `load` then waits
only on Mapterhorn DEM tiles (fast, and pre-warmed by the prefetch path), the
3D view appears in a couple of seconds, and drape tiles stream in
progressively — the same progressive behavior a picker swap already has. The
existing first-`idle` CORS adjudication (`basemapChecked` /
`basemapErrored` / `basemapContentLoaded`) is already gated on `loaded` and
keeps working unchanged; `resolveBasemaps()` still runs before construction
so the picker renders the same list.

**Verification.** `npm run terrain:verify` (real MapLibre frame, hardware
GL, headless) must still pass; add a frame-boot test asserting `loaded` is
posted while a basemap source with permanently-pending tiles is configured
(a fixture tile endpoint that never responds). Manual check per
`docs/development.md` on a BigMap fixture with OpenTopoMap selected.

### B3 — BigMap and Peak pages fail silently; only the analyzer shows an error

On the ascent analyzer, `failTerrain(message)` shows a user-visible message
(`terrainFailureMessage()` maps each bridge reason to actionable copy). The
same failure on BigMap (`big-map.js` `failTerrain()`) and Peak pages
(`peak-map.js` `failTerrain()`) resets the toggle and restores the 2D map
with **no feedback at all**. Combined with B2 this is the reported
experience: a long spinner, then nothing.

**Fix.** Bring the two silent surfaces up to the analyzer's standard: on
`error` (and on the local load timeout), show a small transient status note
near the toggle — same tone and wording as the analyzer's
`terrainFailureMessage()` map, dismissible or auto-hiding, themed like the
toggle. Keep the copy in one place (export the reason→message map from a
shared module rather than pasting it a third time; see I3).

**Regression test.** In `test/maps/big-map.test.mjs` and `test/maps/peak-map.test.mjs`:
dispatch a bridge `error` while loading and assert a visible, accessible
(role=status) message appears and the toggle returns to idle.

### B4 — The user is trapped during a hung load: the toggle is disabled

All three coordinators set `disabled = true` on the toggle while
`terrainState === 'loading'`. With B2's 15–17 s hang, the user stares at a
spinner with no way out. `stopTerrain()` already handles the loading state
correctly (it falls through to `finishTerrainStop()`), so cancel is nearly
free.

**Fix.** Keep the toggle enabled while loading and let a click cancel the
load (label stays "3D"+spinner; title/aria become "Cancel loading 3D
terrain"). Applies to all three surfaces.

### B5 — A drape picked before `load` completes is dropped but marked as chosen

The picker is rendered and interactive during boot, but `swapBasemap()`
returns immediately when `!loaded` — after the `change` handler has already
set `userPickedBasemap = true`. The user's pick never applies, and because
`resumeTerrain()` skips `resolveBasemaps()` when `userPickedBasemap` is set,
a later 2D→3D re-entry also stops following the page's 2D layer — a stale
drape with no way back short of touching the picker again.

**Fix.** Set `userPickedBasemap` only when the swap actually applies (or
queue the pre-load selection and apply it on `load`). B2's restructure
shrinks the pre-load window to almost nothing, but the guard is one line and
closes the state divergence outright.

### B6 — Post-load renderer failures are unhandled (dead black canvas)

The frame's map `error` handler only calls `fail('maplibre')` while
`!loaded`, and nothing listens for WebGL context loss. After `load`, a lost
GL context or fatal renderer error leaves a frozen/black frame the user can
only escape by clicking 2D. The page coordinators are consistent with this
gap: they only accept bridge `error` while `terrainState === 'loading'`.

**Fix.** In the frame, listen for `webglcontextlost` on the map canvas (and
let post-load fatal `error` events through), tear down, and post `error`.
In all three coordinators, accept `error` while `active` too: revert to 2D
via the existing stop path and show the B3 message. Low frequency, cheap
insurance.

## Improvements (not defects)

### I1 — No compass or reset-to-north on the ascent analyzer's 3D view

The frame streams `view` on every surface, but only BigMap and Peak pages
mount `terrainCompass`; the analyzer has no bearing indicator and no
reset-to-north affordance at all (the in-frame NavigationControl
deliberately hides its compass so the control stack height matches 2D).
Once a user rotates on the analyzer they have no one-click way back to
north-up. Mount the shared compass above the analyzer's toggle exactly as
the other two surfaces do.

### I2 — First hover prefetch can be lost to the settings-read race

`terrain-map.js` `forwardPrefetch()` is gated on `terrainEnabled`, which is
false until the async initial `settings.get()` resolves. A hover in that
window drops the warm-up hint. Await `settingsReady` before the gate (the
worker re-validates everything anyway). Minor.

### I3 — Three copy-pasted coordinator state machines invite divergence

`big-map.js`, `peak-map.js`, and the analyzer each hand-roll the same
~200-line terrain state machine (idle/loading/active/stopPending states,
load and camera timers, camera round-trip, message dispatch, toggle
rendering). B3 exists precisely because the copies drifted. After the P1
fixes land, extract a shared page-world `terrain-coordinator` helper that
owns the state machine and failure copy, with each surface supplying its
route/focus collection, mount, and native-map show/hide. Do this as its own
commit series with the existing surface tests as the harness — not bundled
into the bug fixes.

### I4 — Worker-side hygiene was already present

The audit originally reported that `prefetchLastByTab` was not pruned. Current
code already deletes the tab id in the existing `tabs.onRemoved` listener, so
there was no defect to fix and no code change was warranted.

### I5 — Misleading copy for antimeridian routes

`validateRoute()` rejects routes spanning ≥180° of longitude, and the
analyzer maps the resulting `unavailable` to "3D terrain is unavailable in
this browser", which blames the wrong thing. Rare; tweak the copy when the
shared failure-message map (B3/I3) is built.

## Verified sound (do not re-flag)

- `peakRequestBounds()` cos-latitude math is correct for Web Mercator.
- Screen-space peak hit-testing, summit-snap leashes, and the snap-verdict
  cache follow the documented design (`docs/3d-peak-markers.md`).
- The suspend/resume keep-alive protocol (bridge TTL, `resume` payload
  re-application, drape re-resolution) is coherent, including the
  destroy-races-boot path; covered by `test/terrain/terrain-map.test.mjs`.
- Leaflet↔MapLibre camera conversion (±1 zoom for 256 px vs 512 px tiles)
  round-trips correctly.
- The peaks feed client is single-flight with stale-reply suppression; the
  DEM cache is LRU-bounded with fail-open reads.
- The 17 s page timeout deliberately outlasts the 15 s frame timeout so the
  frame fails first with a specific reason.

Ordered by user impact; each phase is an independent, separately-committed
unit with checks run before commit (`AGENTS.md` discipline). No
`manifest.json` or bundle-composition changes are expected before Phase 5,
so `npm run verify:extension` is only mandatory there; run it anyway if a
phase unexpectedly touches load order.

1. **Phase 1 — compass unwrap (B1).** `src/terrain/terrain-compass.js` + regression
   tests in the two surface test files. Checks: `npm test`; visual rotate
   check on the BigMap fixture in a real browser (hidden/protocol-driven per
   the repository's real-browser rules).
2. **Phase 2 — drape no longer gates boot (B2, B5).**
   `src/terrain/terrain-frame.js`: terrain-only constructor style, add drape after
   `load`, guard `userPickedBasemap`. Checks: `npm test`,
   `npm run terrain:verify` (assert hardware renderer as required), new
   pending-drape boot test, manual BigMap+OpenTopoMap fixture check.
3. **Phase 3 — visible failure + cancelable loading (B3, B4).**
   `src/maps/big-map.js`, `src/maps/peak-map.js`, shared failure-copy map, small CSS.
   Checks: `npm test` with the new error-surface tests; visual check of the
   message at BigMap and Peak sizes, light and dark.
4. **Phase 4 — post-load failure handling (B6).** Frame context-loss
   listener; coordinators accept `error` while active. Checks: `npm test`;
   simulate context loss in the terrain:verify harness if practical,
   otherwise document as manually verified via devtools "lose context".
5. **Phase 5 — parity and consolidation (I1, I3, I5, optionally I2).**
   Analyzer compass first (small), then the coordinator extraction as its
   own commit series. Checks: full `npm test`, `npm run terrain:verify`,
   `npm run verify:extension` (bundle composition changes when a new shared
   module is added to `scripts/build-config.mjs`).

After each phase ships, update the terrain sections of
`docs/architecture.md` where behavior changed (drape-after-load, failure
surfaces, compass parity); when the plan is fully implemented or abandoned,
move this file to `docs/archive/` per `docs/plans/README.md`.
